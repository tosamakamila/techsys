一、 根治“伪引导、零推理”：是非题硬禁令与提问框架化
📌 痛点分析
“灵”习惯问“对不对/一样吗”，导致学习者只需回答“是/不是”，随后 AI 接管全盘解释。这是因为 AI 的 Token 预测本能倾向于“尽快给出完整答案”。

🛠 解决方案：行为树（Behavior Tree）约束与负向词汇表
在系统的核心控制 Prompt 中，将“是非题”列为最高级别的违规行为，并强制推行“逆向推导问法”。

Markdown
### 🚫 提问绝对禁令（Negative Constraints）
- 严禁使用任何是非题型。禁止包含词汇：[“是不是”, “对不对”, “一样吗”, “明白了吗”, “懂了吗”, “对吧”]。
- 严禁在学习者未做出关键推理前，输出任何超过 50 字的“概念解释”或“原理总结”。

### 🔄 强制提问范式：过程逆向拆解
当需要让学习者对比 A 和 B 的逻辑时：
1. 错误做法（禁止）：“A 和 B 的底层逻辑是不是一样的？”
2. 正确做法（强制）：
   - 第一步（问过程 A）：“我们先看 A，它是经历哪几个步骤演变成现在这样的？”
   - 第二步（问过程 B）：“那 B 又是怎么变成现在这样的？”
   - 第三步（交出控制权）：“现在把这两个过程摊开放在一起，你发现了什么？”
二、 根治“起点选错、缺乏承接”：显式状态桥接器
📌 痛点分析
AI 频繁在概念间“生硬跳跃”，且直接询问逻辑结果，导致学习者产生断层感，不知道为什么要进入下一个环节。

🛠 解决方案：上下文承接锚点（Bridge Anchor）
在对话管理器的上下文中，强制执行“承接三要素”模板。AI 在切换到新话题前，必须隐式或显式地填满这个模板才能输出下一句话。

承接公式： [回顾已解决的 X] + [引出 X 与 Y 的矛盾/关联] + [开启 Y 的过程提问]

糟糕的跳跃： “刚才说了 Pichia pastoris 的表达，那我们来看看大肠杆菌。”（学习者黑人问号：为什么突然提大肠杆菌？）

标准的承接桥梁： “【回顾】 刚才我们搞清楚了 Pichia pastoris 是通过真核折叠来把这个酶变出来的。【桥梁】 但如果今天我们的宿主没有这种高级的折叠加工厂，比如换成最原始的大肠杆菌，【新起点问过程】 这个酶的合成过程会发生什么变化？”

三、 根治“课后群聊沦为观赏故事”：去剧本化，直写结构化 Insights
📌 痛点分析
原本设计作为诊断机制的群聊，变成了带动作描写的“小说桥段”，对教学优化毫无实质价值。

🛠 解决方案：废除对话生成，改用结构化数据管道
彻底砍掉“群聊剧情”的生成 Prompt，将该模块的 Output Format 直接锁定为向 teaching_insights.md 追加一条干净的 JSON 或 Markdown 诊断日志。

Markdown
### 📥 教学洞察输入规范 (teaching_insights.md)
每次课程结束，系统直接写入以下结构，禁止任何小说式的寒暄或动作描写：

| 维度 | 诊断详情 | 优化策略 (Action Item) |
| :--- | :--- | :--- |
| **推理断层点** | 学习者在 [具体概念/步骤] 处卡壳，未能自主推导。 | 调整该知识点的“过程拆解”步长，增加前置支架。 |
| **交互特征** | 出现 [X] 次含糊回答，表明对 [基础概念] 掌握不牢。 | 在下一次课程前置闪卡中加入相关测试。 |
| **策略有效性**| 使用“过程对比法”后，学习者在 [时间点] 成功自主推导。| 保持该引导路径。 |
四、 根治“下课流程串行阻塞”：主进程挂起与子代理并行化
📌 痛点分析
闪卡生成、课程文件更新等互不依赖的任务串行执行，导致用户在下课时面临超长的等待时间。

🛠 解决方案：异步子代理（Asynchronous Sub-Agents）架构
将主进程降级为一个纯粹的“调度员”（Dispatcher），当收到“下课”指令时，主进程立即执行轻量脚本，同时异步并发触发子任务。

                   [ 收到下课指令 ]
                          |
             ---------------------------
            |                           |
   【主进程：只跑轻量脚本】       【异步并发子代理 (Parallel Tasks)】
    - 关闭当前课程上下文          - Agent A: 提取知识点并生成 Anki 闪卡
    - 给出简短的道别语            - Agent B: 整理并更新课程文件目录
            |                     - Agent C: 提取诊断信息写入 `teaching_insights.md`
            v                           |
   （用户终端立刻恢复可用）                  v
                                  （后台静默完成，不阻塞用户）
五、 如何在代码和 Prompt 层面“卡死”这些设计？
为了确保“灵”在之后的运行中绝对不越界，你需要在它的系统提示词（System Prompt）的开头加入以下重构层面的硬校验关卡：

Markdown
## ⚖️ 运行前置检查门（Pre-flight Checks）
在生成每一句回复前，你必须在后台内核中进行“防御性自检”：
1. **我这句话里包含是非题吗？** 如果有，立刻重写为“询问过程”的句式。
2. **我把结论说出来了吗？** 如果学习者还没说出底层逻辑，我一个字的原理都不准泄露。
3. **两个话题之间有桥梁吗？** 检查是否使用了“刚才搞清楚了...，那...呢？”的承接结构。
4. **我是在写故事吗？** 拒绝任何带有“（微笑着说）”、“（推了推眼镜）”或群聊扮演的废话，保持绝对的教学工具属性。
通过这套“禁令卡死 + 过程提问模板 + 异步去剧情化”的系统设计，可以从根本上把 AI 逼回“引导者”的位置，让学习者被迫进行完整的推理链条。
